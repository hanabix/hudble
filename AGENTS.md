## 测试驱动开发流程

- 每次代码改动前，都确保有对应测试用例可以验证改动的正确性
- 编写或改动测试用例之前，务必与我沟通具体的验证方法，等待确认后再执行
- 每次代码改动后，都要运行测试用例，若出现测试失败，则尝试修复直至所有测试成功
- 运行测试全部成功后，执行代码审查，就其中存在的坏味道提出下一步修改建议

## UI 约束

这是针对单色（#40FF5E）的显示屏的眼镜应用，应：

- 默认情况 UI 的实现应优先遵循 [Rokid YodaOS-Sprite 设计规范](docs/ui-spec.md) 的要求
- 禁止使用 WebView
- 不考虑主题（Theme）
- UI 上的文本优先用英文
- 对于 UI 上的任何调整，需征求确认

## 代码实现

- 采用事件流驱动的异步机制，实现 GATT 的回调处理逻辑
- 在多线程场景下，优先采用只读数据，消息传递，避免共享状态
    - 若存在性能隐患，可采用线程安全的数据类型共享状态，但需征求确认
- 原则上，变量和参数的命名在上下文唯一的情况下，尽可能简短
    - 应避免类型信息的重复，如 `val serviceUuid: UUID` 应为 `val service: UUID`
- 代码注解优先用简短的英文
- 富有业务含义的类型优于基础类型

## 测试约束

- 在没有可用设备的时候，禁止执行 `connectedDebugAndroidTest` 测试
